    Flujo de trabajo

    flowchart TD A[Editar .md en repo privado
    veslava-edu/dwes-md] --> B[git add/commit/push] B --> C[GitHub Actions
    Workflow publish.yml] C --> D[Instalar deps + mkdocs build] D --> E[Generar sitio en /site] E --> F[Deploy a repo público
    veslava-edu/dwes: gh-pages] F --> G[GitHub Pages sirve
    https://veslava-edu.github.io/dwes/]

    C -->|Fallo| CERR(Build/Deploy falla)
CERR -->|Revisar logs| C

    sequenceDiagram autonumber participant Dev as Tú (local) participant MD as Repo privado
    dwes-md participant GA as GitHub Actions
    publish.yml participant PUB as Repo público
    dwes (gh-pages) participant GP as GitHub Pages

    Dev->>MD: git push (cambios .md)
MD-->>GA: Dispara workflow (on: push)
GA->>GA: Instalar deps (mkdocs/material, etc.)
GA->>GA: mkdocs build → genera /site
GA->>PUB: push a rama gh-pages (actions-gh-pages)
PUB-->>GP: Actualiza contenido publicado
GP-->>Dev: Sitio disponible en /dwes/

    Puntos de control: 1. Commit & push en dwes-md → comprueba que el push llega a GitHub. 2. Acción de GitHub → revisar si el workflow publish.yml se ejecuta sin errores. 3. Build de mkdocs → si falla, revisar dependencias o mkdocs.yml. 4. Deploy a gh-pages → si no aparece nada nuevo en dwes, revisar permisos del token. 5. GitHub Pages → comprobar que la URL carga y los enlaces funcionan.

    Errores comunes: - ❌ No hay upstream branch: ocurre si no configuraste aún el remoto. Solución: ```bash git push --set-upstream origin main
