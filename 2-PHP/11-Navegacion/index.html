    Navegación en PHP: Creando el GPS de tu Aplicación Web

    ¡Hola, futuros cracks del desarrollo web! Hoy vamos a hablar de algo fundamental: la navegación. Si una aplicación web fuera una ciudad, la navegación serían sus calles, avenidas y su sistema de GPS. Es lo que permite a los usuarios moverse de un punto a otro, desde la página de inicio hasta el perfil de un amigo, desde una lista de productos hasta la pasarela de pago. Sin una buena navegación, los usuarios se pierden, se frustran y, como en un videojuego mal diseñado, abandonan la partida.

    Nuestra misión como desarrolladores del lado del servidor con PHP es ser los arquitectos de estas rutas. No solo creamos las páginas (los edificios), sino que también diseñamos las conexiones inteligentes que las unen, decidiendo qué información viaja de un punto a otro y cómo lo hace. ¿Listos para poneros el casco de urbanista digital? ¡Vamos a ello!

    Conceptos Fundamentales

    Antes de asfaltar nuestras primeras rutas, aclaremos algunos términos clave. Son los cimientos sobre los que construiremos toda la lógica de navegación.

    Definición: Petición HTTP

    Es el mensaje que un cliente (normalmente un navegador web) envía a un servidor para solicitar un recurso (como una página web, una imagen o datos). Es el "Oye, servidor, ¿me das esto, por favor?".

    Definición: Respuesta HTTP

    Es el paquete que el servidor devuelve al cliente tras recibir y procesar una petición HTTP. Contiene el recurso solicitado (por ejemplo, el código HTML de la página) y un código de estado que indica si todo ha ido bien.

    Definición: Superglobales de PHP

    Son variables especiales, predefinidas en PHP, que están disponibles en todos los ámbitos a lo largo del script. Para la navegación, las más importantes son $_GET, $_POST, y $_SERVER, que nos permiten acceder a la información enviada desde el cliente. ¡Son como el panel de control de nuestro coche!

    Definición: URL (Uniform Resource Locator)

    Es la dirección única que identifica un recurso en la web. Se compone de varias partes: protocolo (https), dominio (google.com), ruta (/search) y, a veces, una cadena de consulta o query string (?q=navegacion+php), que es crucial para pasar información.

    Desarrollo y Ejemplos Prácticos

    Ahora que tenemos el mapa, vamos a explorar los tres caminos principales para guiar a nuestros usuarios por la aplicación.

    Navegación mediante Enlaces: El método "Point-and-Click"

    La forma más básica y directa de navegación. Es el equivalente a seguir las señales de tráfico en una ciudad. Se implementa con la etiqueta <a> de HTML, pero el verdadero poder para nosotros, como desarrolladores PHP, está en cómo le añadimos "información extra" a esa señal.

    Esto se logra a través de la cadena de consulta (o query string) en la URL, que empieza con un ?. Esta es la manera de enviar datos mediante el método GET.

    Ejemplo 1: Navegación simple entre un blog y su único post

    Imaginemos un blog muy minimalista. Desde el index.php queremos enlazar a un post.php.

    <!-- index.php -->
<!DOCTYPE html>
<html lang="es">
<head>
    <meta charset="UTF-8">
    <title>Mi Blog Minimalista</title>
</head>
<body>
    <h1>Bienvenido a mi Blog</h1>
    <p>Solo tengo un artículo, pero es muy bueno.</p>
    <a href="post.php">Leer el único post</a>
</body>
</html>

    <!-- post.php -->
<!DOCTYPE html>
<html lang="es">
<head>
    <meta charset="UTF-8">
    <title>Mi Gran Post</title>
</head>
<body>
    <h1>El arte de la programación</h1>
    <p>Programar es como un superpoder, ¡úsalo con responsabilidad!</p>
    <a href="index.php">Volver al inicio</a>
</body>
</html>

    Ejemplo 2: Pasando datos con GET para un blog con múltiples posts

    Aquí es donde PHP empieza a brillar. En lugar de una página por cada post, tenemos una única plantilla (post.php) que muestra el contenido correcto según un id que le pasamos por la URL.

    <!-- index.php -->
<?php
// En una aplicación real, estos datos vendrían de una base de datos.
$posts = [
    1 => ['titulo' => 'Introducción a PHP 8+', 'autor' => 'Ana'],
    2 => ['titulo' => '¿Por qué PSR-12 es tu amigo?', 'autor' => 'Luis'],
    3 => ['titulo' => 'El patrón PRG explicado con gatitos', 'autor' => 'Carla']
];
?>
<!DOCTYPE html>
<html lang="es">
<head>
    <meta charset="UTF-8">
    <title>Mi Blog con Varios Posts</title>
</head>
<body>
    <h1>Últimos Artículos</h1>
    <ul>
        <?php foreach ($posts as $id => $post): ?>
            <li>
                <a href="post.php?id=<?= (int)$id ?>">
                    <?= htmlspecialchars($post['titulo']) ?>
                </a>
            </li>
        <?php endforeach; ?>
    </ul>
</body>
</html>

    <!-- post.php -->
<?php
// Simulamos de nuevo una base de datos.
$posts = [
    1 => ['titulo' => 'Introducción a PHP 8+', 'contenido' => 'PHP 8 ha traído muchas novedades...'],
    2 => ['titulo' => '¿Por qué PSR-12 es tu amigo?', 'contenido' => 'Mantener un código limpio es crucial...'],
    3 => ['titulo' => 'El patrón PRG explicado con gatitos', 'contenido' => 'Imagina que un gato envía un formulario...']
];

// Obtenemos el ID de la URL. ¡Siempre validar y sanitizar!
$postId = isset($_GET['id']) ? (int)$_GET['id'] : 0;

// Verificamos si el post existe.
if (!isset($posts[$postId])) {
    // Si no existe, podemos mostrar un error o redirigir.
    header("Location: 404.php");
    exit;
}

$post = $posts[$postId];
?>
<!DOCTYPE html>
<html lang="es">
<head>
    <meta charset="UTF-8">
    <title><?= htmlspecialchars($post['titulo']) ?></title>
</head>
<body>
    <h1><?= htmlspecialchars($post['titulo']) ?></h1>
    <p><?= htmlspecialchars($post['contenido']) ?></p>
    <hr>
    <a href="index.php">Volver al listado de artículos</a>
</body>
</html>

    URLs amigables y SEO

    Habrás notado que muchas webs usan URLs como /post/el-titulo-del-post en lugar de post.php?id=2. Esto se conoce como "URL amigable" y es mejor para el SEO y la experiencia de usuario. Se consigue con reescritura de URLs en el servidor (usando .htaccess en Apache o nginx.conf), pero por debajo, el mecanismo sigue siendo el mismo: el servidor traduce esa URL bonita a una que PHP pueda entender, como post.php?slug=el-titulo-del-post.

    Preguntas para reflexionar

    1. ¿Por qué es una pésima idea de seguridad enviar un dato sensible como una contraseña a través de un parámetro GET en la URL?
    2. Si estuvieras creando una galería de fotos, ¿cómo usarías los parámetros GET para implementar los botones "Siguiente" y "Anterior"?
    3. Entra en una web de e-commerce, navega por las categorías y aplica un filtro (por ejemplo, por precio o marca). Observa la URL. ¿Qué parámetros GET puedes identificar y qué crees que hace cada uno?
    4. ¿Por qué en el ejemplo 2 usamos (int)$_GET['id'] y htmlspecialchars()? ¿Qué riesgos de seguridad estamos mitigando?

    Navegación mediante Formularios: El Interrogatorio Digital

    Los formularios son la herramienta principal para que el usuario nos envíe información más compleja o sensible. Piénsalo como rellenar un documento para solicitar algo: un inicio de sesión, un registro, una búsqueda, etc. La etiqueta <form> tiene dos atributos clave: action (el script PHP que procesará los datos) y method (cómo se enviarán los datos). Aquí es donde la batalla GET vs POST cobra protagonismo.

    Característica Método GET Método POST
    Visibilidad de datos Los datos se añaden a la URL (visibles, inseguros para datos sensibles). Los datos se envían en el cuerpo de la petición HTTP (ocultos).
    Límite de datos Limitado por la longitud máxima de la URL (aprox. 2000 caracteres). Prácticamente sin límite, ideal para grandes cantidades de datos o archivos.
    Marcadores/Historial La URL resultante se puede guardar en marcadores y queda en el historial. No se puede guardar en marcadores. Recargar la página puede reenviar los datos.
    Uso principal Solicitar datos del servidor (búsquedas, filtros, paginación). Enviar datos para crear o modificar un recurso (registros, logins, subir archivos).
    Superglobal PHP $_GET $_POST

    Ejemplo 1: Un formulario de búsqueda (usando GET)

    Un formulario de búsqueda es el caso de uso perfecto para GET. La búsqueda es una consulta, y el resultado es una página que el usuario podría querer guardar o compartir.

    <!-- index.php -->
<!DOCTYPE html>
<html lang="es">
<head>
    <meta charset="UTF-8">
    <title>Buscador de Personajes</title>
</head>
<body>
    <h1>Buscador de Personajes de Videojuegos</h1>
    <form action="buscar.php" method="GET">
        <label for="termino">Buscar:</label>
        <input type="text" id="termino" name="q" placeholder="Ej: Mario, Lara Croft...">
        <input type="submit" value="Buscar">
    </form>
</body>
</html>

    <!-- buscar.php -->
<?php
// En un caso real, haríamos una consulta a la base de datos con el término de búsqueda.
$personajes = ['Mario', 'Lara Croft', 'Master Chief', 'Zelda', 'Geralt de Rivia'];
$resultados = [];
$terminoBusqueda = '';

if (isset($_GET['q']) && !empty(trim($_GET['q']))) {
    $terminoBusqueda = trim($_GET['q']);
    foreach ($personajes as $personaje) {
        // Búsqueda simple insensible a mayúsculas/minúsculas
        if (stripos($personaje, $terminoBusqueda) !== false) {
            $resultados[] = $personaje;
        }
    }
}
?>
<!DOCTYPE html>
<html lang="es">
<head>
    <meta charset="UTF-8">
    <title>Resultados de Búsqueda</title>
</head>
<body>
    <h1>Resultados para "<?= htmlspecialchars($terminoBusqueda) ?>"</h1>

    <?php if (!empty($resultados)): ?>
        <ul>
            <?php foreach ($resultados as $resultado): ?>
                <li><?= htmlspecialchars($resultado) ?></li>
            <?php endforeach; ?>
        </ul>
    <?php elseif (!empty($terminoBusqueda)): ?>
        <p>No se encontraron personajes con ese nombre. ¡Quizás en otro castillo!</p>
    <?php endif; ?>

    <p><a href="index.php">Realizar otra búsqueda</a></p>
</body>
</html>

    Ejemplo 2: Un formulario de contacto (usando POST)

    Para un formulario de contacto, donde el usuario envía datos personales, POST es la única opción sensata. No queremos que su email y mensaje aparezcan en la URL.

    <!-- contacto.php -->
<!DOCTYPE html>
<html lang="es">
<head>
    <meta charset="UTF-8">
    <title>Contacto</title>
</head>
<body>
    <h1>Formulario de Contacto</h1>
    <form action="procesar_contacto.php" method="POST">
        <p>
            <label for="email">Tu Email:</label><br>
            <input type="email" id="email" name="email" required>
        </p>
        <p>
            <label for="mensaje">Mensaje:</label><br>
            <textarea id="mensaje" name="mensaje" rows="5" required></textarea>
        </p>
        <p>
            <input type="submit" value="Enviar Mensaje">
        </p>
    </form>
</body>
</html>

<?php
// Comprobamos si el formulario fue enviado usando el método POST.
if ($_SERVER['REQUEST_METHOD'] === 'POST') {
    // Es crucial validar y limpiar los datos del usuario.
    $email = filter_input(INPUT_POST, 'email', FILTER_VALIDATE_EMAIL);
    $mensaje = filter_input(INPUT_POST, 'mensaje', FILTER_SANITIZE_STRING);

    if ($email && $mensaje) {
        // Aquí iría la lógica para enviar el email.
        // mail($emailAdmin, "Nuevo mensaje de contacto", $mensaje);
        $respuesta = "Gracias por tu mensaje, " . htmlspecialchars($email) . ". Te responderemos pronto.";
    } else {
        $respuesta = "Datos inválidos. Por favor, vuelve atrás y rellena el formulario correctamente.";
    }
} else {
    // Si alguien accede a este script directamente, lo mandamos al formulario.
    header('Location: contacto.php');
    exit;
}
?>
<!DOCTYPE html>
<html lang="es">
<head>
    <meta charset="UTF-8">
    <title>Mensaje Enviado</title>
</head>
<body>
    <h1>Estado del Envío</h1>
    <p><?= $respuesta ?></p>
    <a href="contacto.php">Enviar otro mensaje</a>
</body>
</html>

    ¡Nunca confíes en los datos del usuario!

    Este es el mandamiento número uno del desarrollo web. SIEMPRE valida y sanea cualquier dato que venga del cliente ($_GET, $_POST, $_COOKIE, etc.). Usa funciones como filter_input(), htmlspecialchars(), strip_tags() y sentencias preparadas en tus consultas a la base de datos para prevenir vulnerabilidades como XSS (Cross-Site Scripting) y la Inyección SQL.

    Preguntas para reflexionar

    1. Cuando rellenas un formulario POST y le das a "Refrescar" al navegador, a menudo te sale una advertencia del tipo "¿Desea volver a enviar la información del formulario?". ¿Por qué ocurre esto con POST y no con GET?
    2. Si tuvieras que implementar un botón de "Eliminar" para un artículo de un blog, ¿deberías usar un enlace (<a>) o un formulario (<form>)? ¿Qué método HTTP sería el más apropiado y por qué?
    3. La superglobal $_REQUEST contiene los datos de $_GET, $_POST y $_COOKIE. ¿Por qué crees que muchos desarrolladores consideran una mala práctica usar $_REQUEST en lugar de las superglobales específicas?


    Navegación mediante Redirecciones: El Teletransporte del Servidor

    Una redirección es cuando el servidor le dice al navegador: "Oye, lo que buscas no está aquí, vete a esta otra URL". Es una orden directa. En PHP, esto se hace con la función header().

    La regla de oro: la función header() DEBE ser llamada antes de que se envíe cualquier tipo de salida al navegador. Esto incluye etiquetas HTML, espacios en blanco antes de <?php, o incluso un echo. Si no, te encontrarás con el temido error: "Warning: Cannot modify header information - headers already sent".

    El Patrón PRG: Post/Redirect/Get

    Este es uno de los patrones de diseño más importantes en el desarrollo web. Soluciona el problema del reenvío de formularios POST que comentábamos antes. El flujo es el siguiente: 1. POST: El usuario envía un formulario con el método POST. 2. Redirect: El servidor procesa los datos (por ejemplo, los guarda en la base de datos). En lugar de mostrar una página de confirmación directamente, envía una respuesta de redirección al navegador, indicándole que vaya a una nueva URL (la página de éxito o de vuelta al formulario). 3. GET: El navegador recibe la orden de redirección y hace una nueva petición GET a la URL indicada. Ahora está en una página "segura" que puede recargar sin problemas.

    Código HTTP Nombre Cuándo usarlo
    301 Moved Permanently (Movido Permanentemente) Para indicar a los navegadores y a los motores de búsqueda que una página se ha movido a una nueva URL para siempre. Es crucial para el SEO, ya que traspasa el "valor" de la página antigua a la nueva.
    302 Found (Encontrado) / 303 See Other Para redirecciones temporales. El patrón PRG es el caso de uso perfecto. Le dices al navegador que la acción POST fue exitosa y que ahora debe ver (GET) otra página. Por defecto, header('Location: ...') envía un 302.

    Ejemplo 1: Redirección simple para una zona privada

    Si un usuario intenta acceder a dashboard.php sin haber iniciado sesión, lo redirigimos a login.php.

    // dashboard.php
<?php
session_start(); // Necesitamos sesiones para saber si está logueado

// Si no existe la variable de sesión 'usuario_id', no ha iniciado sesión
if (!isset($_SESSION['usuario_id'])) {
    // Redirigimos a la página de login
    header('Location: login.php');
    // Es VITAL detener la ejecución del script después de una redirección
    exit;
}

// Si el script llega hasta aquí, es que el usuario sí está autenticado.
?>
<!DOCTYPE html>
<html>
<head><title>Mi Panel de Control</title></head>
<body>
    <h1>¡Bienvenido a tu Dashboard!</h1>
    <p>Aquí tienes tus cosas super secretas.</p>
</body>
</html>

    Ejemplo 2: Implementando el patrón PRG

    Este es el ejemplo del formulario de contacto, pero bien hecho, aplicando el patrón PRG para evitar el reenvío de datos.

    // procesar_y_redirigir_contacto.php
<?php
session_start();

if ($_SERVER['REQUEST_METHOD'] === 'POST') {
    $email = filter_input(INPUT_POST, 'email', FILTER_VALIDATE_EMAIL);
    $mensaje = filter_input(INPUT_POST, 'mensaje', FILTER_SANITIZE_STRING);

    if ($email && $mensaje) {
        // Lógica de envío de email...
        // Guardamos un mensaje de éxito en la sesión (flash message)
        $_SESSION['flash_message'] = "¡Mensaje enviado con éxito! Gracias.";
        $_SESSION['flash_type'] = "success";
    } else {
        $_SESSION['flash_message'] = "Error: los datos proporcionados no son válidos.";
        $_SESSION['flash_type'] = "error";
    }

    // ¡La magia del PRG! Redirigimos de vuelta al formulario.
    header('Location: contacto_con_prg.php');
    exit;
}

<?php
session_start();

// Comprobamos si hay un "flash message" para mostrar
$flashMessage = null;
if (isset($_SESSION['flash_message'])) {
    $flashMessage = [
        'message' => $_SESSION['flash_message'],
        'type' => $_SESSION['flash_type']
    ];
    // Lo borramos para que no se muestre de nuevo si recargan la página
    unset($_SESSION['flash_message']);
    unset($_SESSION['flash_type']);
}
?>
<!DOCTYPE html>
<html lang="es">
<head>
    <meta charset="UTF-8">
    <title>Contacto (con PRG)</title>
    <style>
        .message { padding: 10px; margin-bottom: 15px; border-radius: 5px; }
        .success { background-color: #d4edda; color: #155724; }
        .error { background-color: #f8d7da; color: #721c24; }
    </style>
</head>
<body>
    <h1>Formulario de Contacto Mejorado</h1>

    <?php if ($flashMessage): ?>
        <div class="message <?= htmlspecialchars($flashMessage['type']) ?>">
            <?= htmlspecialchars($flashMessage['message']) ?>
        </div>
    <?php endif; ?>

    <form action="procesar_y_redirigir_contacto.php" method="POST">
        <!-- ... campos del formulario ... -->
    </form>
</body>
</html>

    exit; es tu mejor amigo

    Lo repetimos porque es causa de incontables bugs y problemas de seguridad: después de llamar a header('Location: ...'), siempre llama a exit; o die();. Si no lo haces, el script de PHP continuará ejecutándose. Un atacante podría, en ciertas circunstancias, saltarse la redirección y ejecutar código que no debería.

    Preguntas para reflexionar

    1. Una empresa cambia su nombre de dominio de viejaweb.com a nuevaweb.com. ¿Qué tipo de redirección (301 o 302) deberían usar en el servidor antiguo y por qué es tan importante para su posicionamiento en Google?
    2. ¿Qué crees que pasaría en el ejemplo 2 (contacto_con_prg.php) si nos olvidáramos de hacer unset() a las variables de sesión del flash_message?
    3. Piensa en el flujo de un carrito de la compra. ¿En qué momentos específicos crees que se utiliza el patrón Post/Redirect/Get?
    4. ¿Por qué crees que el error "headers already sent" es uno de los más comunes para los programadores que empiezan con PHP?


    Aplicación en el Mundo Real

    Estas tres técnicas de navegación no son conceptos teóricos; son el pan de cada día en cualquier aplicación web que uses.

    • Amazon o cualquier E-commerce: Cuando haces clic en una categoría de productos, estás usando un enlace con parámetros GET (/category?id=electronica). Cuando aplicas un filtro de precio, añades más parámetros GET (...&price_min=100&price_max=500). Cuando añades un producto al carrito, envías un formulario POST. Cuando finalizas la compra, el sistema procesa tu pago (POST) y te redirige a una página de "Gracias por tu compra".
    • WordPress: Cuando editas una entrada y pulsas "Actualizar", estás enviando un POST con todo el contenido. Una vez guardado, WordPress te redirige de nuevo al editor o a la lista de entradas, a menudo con un parámetro GET en la URL (?message=1) para mostrarte el aviso "Entrada actualizada".
    • Redes Sociales: Al hacer clic en el perfil de un usuario, es una navegación por enlace. Al publicar un nuevo estado o subir una foto, es un formulario POST. Inmediatamente después, la plataforma te redirige o actualiza la vista para que veas tu nueva publicación (hoy en día, esto se hace de forma más fluida con JavaScript y AJAX, pero el patrón PRG sigue siendo la base de la operación en el servidor).

    Para Saber Más

    Si quieres clavar estos conceptos, te recomiendo encarecidamente que explores estos recursos. ¡La curiosidad es el motor del buen programador!

    • Documentación oficial de PHP sobre header(): https://www.php.net/manual/es/function.header.php - La fuente de la verdad. Entender todos sus parámetros te dará un control total sobre las cabeceras HTTP.
    • MDN Web Docs sobre Métodos de Petición HTTP: https://developer.mozilla.org/es/docs/Web/HTTP/Methods - Una explicación completísima y clara de GET, POST y otros métodos HTTP desde la perspectiva del navegador.
    • Artículo sobre el Patrón Post-Redirect-Get (PRG): https://www.theserverside.com/news/1365146/Redirect-After-Post (en inglés) - Un artículo técnico que profundiza en el porqué y el cómo del patrón PRG, un pilar de las aplicaciones web robustas.

    2) Formulario de Registro a Workshop | Nivel: Fácil

    Ejercicios de Refuerzo

    3) Evolución a PRG: El Registro Anti-Duplicados | Nivel: Medio

    Objetivo:

    Refactorizar una implementación de formulario POST para aplicar el patrón Post/Redirect/Get (PRG), utilizando redirecciones y "flash messages" con sesiones para mejorar la experiencia de usuario y la robustez de la aplicación.

    Setup Inicial:

    Partimos del ejercicio anterior (register_workshop.php y process_registration.php). El sistema funciona, pero tiene un fallo clásico: si un usuario envía el formulario y luego refresca la página de "Registro Exitoso", el navegador intentará reenviar los datos, creando un registro duplicado. ¡Hay que solucionarlo!

    Tarea a realizar:
    1. Activa las sesiones: En la parte superior de ambos scripts (register_workshop.php y process_registration.php), asegúrate de iniciar la sesión con session_start().
    2. Modifica process_registration.php:
      • Después de validar los datos, en lugar de generar un HTML, guarda el mensaje de éxito o error en una variable de sesión (por ejemplo, $_SESSION['flash_message']).
      • Una vez guardado el mensaje, redirige siempre al usuario de vuelta a register_workshop.php usando header('Location: register_workshop.php');. ¡No olvides el exit;!

    3. Modifica register_workshop.php:

      • Al principio del script, comprueba si existe la variable de sesión $_SESSION['flash_message'].
      • Si existe, muestra su contenido en un lugar visible de la página (por ejemplo, encima del formulario).
      • Muy importante: Después de mostrar el mensaje, bórralo de la sesión (unset($_SESSION['flash_message']);) para que no vuelva a aparecer si el usuario recarga la página de nuevo.

    4. Pista: Piensa en la sesión como un "buzón de mensajes" temporal entre dos peticiones. El script que procesa el POST deja una carta en el buzón y el script del formulario la recoge y la muestra, tirándola después a la basura.

    Aplicación en el Mundo Real:

    El patrón PRG es un estándar de la industria para manejar envíos de formularios que modifican datos. Se utiliza en todas partes: al publicar un comentario, al actualizar un perfil, al realizar una compra online, etc. Dominarlo es esencial para crear aplicaciones web que se comporten de forma predecible y profesional.

    4) Reto de Búsqueda Avanzada | Nivel: Alto

    Objetivo:

    Diseñar e implementar una funcionalidad de búsqueda y filtrado compleja, combinando formularios con el método GET, construcción dinámica de URLs y procesamiento de múltiples parámetros de entrada. Fomentar la colaboración y la crítica constructiva a través de una revisión por pares.

    Setup Inicial:

    "EventFlow" quiere ofrecer a sus asistentes una potente página de búsqueda para explorar el programa de la conferencia. Necesitan poder filtrar las charlas por temática (Track), por ponente y por un término de búsqueda libre en el título.

    Datos base (en search.php): 

    <?php
$talks = [
    ['id' => 'ai-101', 'title' => 'IA Generativa para Desarrolladores', 'speaker' => 'Dra. Ada Lovelace', 'track' => 'AI & ML'],
    ['id' => 'sec-202', 'title' => 'Defensa contra el Lado Oscuro de la Web (XSS)', 'speaker' => 'Prof. Alan Turing', 'track' => 'Cybersecurity'],
    ['id' => 'php-303', 'title' => 'Novedades Asíncronas en PHP 8.3', 'speaker' => 'Sr. Rasmus Lerdorf', 'track' => 'Backend Development'],
    ['id' => 'ai-201', 'title' => 'Ética en el Desarrollo de IA', 'speaker' => 'Dra. Ada Lovelace', 'track' => 'AI & ML'],
    ['id' => 'be-401', 'title' => 'Arquitecturas Serverless con PHP', 'speaker' => 'Sr. Rasmus Lerdorf', 'track' => 'Backend Development']
];

// Extraer opciones únicas para los filtros <select>
$speakers = array_unique(array_column($talks, 'speaker'));
$tracks = array_unique(array_column($talks, 'track'));
?>

    Tarea a realizar:
    1. Crea search.php: Este fichero contendrá tanto el formulario de búsqueda como la lógica de filtrado y la visualización de resultados.
    2. El Formulario (método GET):
      • Un campo de texto para buscar por término (q).
      • Un menú desplegable (<select>) para filtrar por ponente (speaker). Debe tener una opción por defecto "Todos los ponentes".
      • Un menú desplegable (<select>) para filtrar por track (track). Debe tener una opción por defecto "Todas las temáticas".
    3. La Lógica de Filtrado (en el mismo fichero):
      • Recupera los parámetros q, speaker y track de la superglobal $_GET.
      • Comienza con la lista completa de $talks y ve aplicando los filtros secuencialmente. Si un filtro no se ha especificado (o su valor es "todos"), no se debe aplicar.
      • La búsqueda por término (q) debe ser insensible a mayúsculas/minúsculas y buscar dentro del title de la charla.
    4. Visualización:
      • Muestra los filtros seleccionados actualmente encima de los resultados.
      • Muestra la lista de charlas que cumplen los criterios de búsqueda. Si no hay resultados, muestra un mensaje amigable.
    5. Revisión por Pares (Peer Review):
      • Intercambia tu solución (search.php) con un compañero.
      • Evalúa su código basándote en los siguientes criterios:
        • Funcionalidad: ¿Funcionan correctamente todos los filtros, tanto individualmente como combinados?
        • Seguridad: ¿Se utiliza htmlspecialchars() en toda la salida de datos para prevenir XSS?
        • Experiencia de Usuario (UX): ¿El formulario "recuerda" los valores seleccionados después de enviarse? (Pista: necesitas usar PHP para añadir el atributo selected a las opciones <option> correctas).
        • Claridad del Código: ¿El código es legible y está bien comentado?
      • Proporciona feedback constructivo a tu compañero.
    Aplicación en el Mundo Real:

    Las páginas de búsqueda y filtrado son el corazón de las aplicaciones ricas en contenido como e-commerce (filtrar por marca, precio, talla), portales de noticias o plataformas de streaming. Construirlas de manera eficiente y segura es una tarea de desarrollo backend de alto valor.
