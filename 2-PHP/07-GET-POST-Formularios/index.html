    PHP: GET, POST y el Arte de Conversar con Formularios

    ¡Hola, futuros cracks del desarrollo web! Hoy vamos a sumergirnos en uno de los pilares de cualquier aplicación web interactiva: los formularios. Si alguna vez te has registrado en una web, has comentado en un blog o has comprado algo online, ya has estado en el lado del cliente de esta conversación. Ahora, vamos a aprender a ser el servidor que escucha, entiende y responde.

    La comunicación entre un navegador y un servidor a través de formularios es como pedir comida en un restaurante muy peculiar. Tienes dos maneras de hacer tu pedido:

    1. El Método GET: Es como gritar tu pedido desde la mesa al cocinero. "¡QUIERO UNA HAMBURGUESA CON QUESO Y SIN PEPINILLOS!". Es rápido y directo, pero todo el restaurante se entera de lo que has pedido. Tu pedido (los datos) forma parte de la propia dirección (la URL), es visible para todos y tiene un límite en lo que puedes gritar.
    2. El Método POST: Es como escribir tu pedido en una nota, doblarla y que el camarero la lleve discretamente a la cocina. Nadie ve el contenido del pedido durante el trayecto. Puedes escribir una comanda larguísima (enviar muchos datos, incluso una foto de cómo quieres el plato), y es mucho más privado.

    En este tema, aprenderemos a ser tanto el cliente que escribe la nota como el cocinero que la recibe, la procesa y prepara el plato. ¡Vamos al lío!


    Conceptos Fundamentales

    Antes de empezar a tirar código, aseguremos las bases. Estos son los términos que usarás a diario.

    Definición: Cliente

    Es el que inicia la conversación. En el 99% de los casos, será el navegador web (Chrome, Firefox, Safari...) que una persona usa en su ordenador, tablet o móvil. Es el que tiene una necesidad y "pide" algo al servidor.

    Definición: Servidor

    Es la máquina (o conjunto de máquinas) que está a la escucha, esperando peticiones de los clientes. Nuestro código PHP vivirá aquí. Su trabajo es recibir la petición, procesarla (quizás consultando una base de datos) y devolver una respuesta, normalmente una página HTML.

    Definición: Petición HTTP

    Es el mensaje formal que envía el cliente al servidor. Imagina un sobre que contiene la dirección del destinatario (el servidor), el remitente y, lo más importante, qué es lo que quiere. Esta petición se hace usando principalmente dos "métodos": GET y POST.

    Definición: Superglobales de PHP

    Son variables especiales que PHP crea automáticamente y que están disponibles en cualquier parte de tu código. Para los formularios, son nuestro "buzón de entrada". Las principales son $_GET, $_POST, $_REQUEST y $_FILES. Contienen los datos que el cliente nos envía.


    Desarrollo y Ejemplos Prácticos

    Creando tu Primer Formulario: La Etiqueta <form>

    Todo empieza con HTML. La etiqueta <form> es el contenedor de todos los campos que el usuario rellenará. Sus dos atributos más importantes son action y method.

    • action: Define a qué script PHP (a qué URL) se enviarán los datos cuando el usuario pulse el botón de enviar. Si lo dejas en blanco, los datos se enviarán al mismo script que contiene el formulario.
    • method: Especifica cómo se enviarán los datos. Sus valores son get o post.
    <!-- Archivo: formulario_saludo.html -->
<!DOCTYPE html>
<html lang="es">
<head>
    <meta charset="UTF-8">
    <title>Formulario de Saludo</title>
</head>
<body>
    <form action="procesar_saludo.php" method="post">
        <label for="nombre">Introduce tu nombre:</label>
        <input type="text" id="nombre" name="usuario_nombre">

        <input type="submit" value="Enviar Saludo">
    </form>
</body>
</html>

    Este es nuestro punto de partida. Un formulario simple que pide un nombre y, al pulsar "Enviar Saludo", enviará los datos al fichero procesar_saludo.php usando el método POST.


    El Método GET: Datos a la Vista de Todos

    El método GET envía los datos del formulario como una cadena de pares nombre=valor añadida al final de la URL en el atributo action.

    Características clave:

    • Visibilidad: Los datos son visibles en la barra de direcciones del navegador.
    • Marcadores: Como los datos están en la URL, los usuarios pueden guardar la página resultante en sus marcadores.
    • Límites: Tiene un límite de longitud (unos 2000 caracteres). No es apto para enviar grandes cantidades de datos.
    • Idempotencia: Realizar la misma petición GET múltiples veces debería producir el mismo resultado.

    ¡Cuidado con el GET!

    Nunca, jamás, uses GET para enviar información sensible como contraseñas, datos personales o cualquier cosa que no quieras que quede registrada en el historial del navegador o en los logs del servidor.

    Ejemplo 1: Un buscador simple

    Imagina un buscador para tu blog. Usar GET es ideal aquí, porque los usuarios podrían querer guardar la URL de una búsqueda específica.

    HTML (buscador.html): 

    <form action="buscar.php" method="get">
    <label for="termino">Buscar en el blog:</label>
    <input type="search" id="termino" name="q">
    <button type="submit">Buscar</button>
</form>

    PHP (buscar.php): 

    <?php
// Siempre comprueba si la variable existe antes de usarla
if (isset($_GET['q']) && !empty($_GET['q'])) {
    // Sanitizamos la entrada para evitar ataques XSS
    $termino_busqueda = htmlspecialchars($_GET['q']);
    echo "<h1>Resultados de la búsqueda para: " . $termino_busqueda . "</h1>";
    // Aquí iría la lógica para buscar en la base de datos...
} else {
    echo "<h1>Por favor, introduce un término de búsqueda.</h1>";
}
?>
    Si un usuario busca "PHP es genial", la URL resultante será: http://tusitio.com/buscar.php?q=PHP+es+genial.

    Ejemplo 2: Paginación de resultados

    GET es perfecto para cosas como la paginación, donde quieres navegar a una página específica.

    PHP (lista_productos.php): 

    <?php
// Asumimos que la página actual viene por GET. Si no, es la página 1.
$pagina_actual = isset($_GET['pagina']) ? (int)$_GET['pagina'] : 1;

echo "<h2>Mostrando productos de la página: " . htmlspecialchars($pagina_actual) . "</h2>";
// Lógica para mostrar los productos...

// Generar enlaces a otras páginas
$pagina_siguiente = $pagina_actual + 1;
$pagina_anterior = $pagina_actual - 1;

echo "<nav>";
if ($pagina_anterior > 0) {
    echo "<a href='lista_productos.php?pagina={$pagina_anterior}'>Anterior</a> ";
}
echo "<a href='lista_productos.php?pagina={$pagina_siguiente}'>Siguiente</a>";
echo "</nav>";
?>

    Reflexiona

    1. Aparte de buscadores y paginación, ¿en qué otra situación del mundo real crees que el método GET sería la mejor opción?
    2. Si creas una galería de imágenes y quieres que los usuarios puedan enlazar directamente a la foto número 5, ¿qué método usarías y cómo construirías la URL?
    3. ¿Qué riesgos de seguridad, aparte de la visibilidad de los datos, puede presentar el uso inadecuado de GET?


    El Método POST: El Estándar para la Entrada de Datos

    POST es el método de elección para la mayoría de los formularios. Envía los datos en el cuerpo de la petición HTTP, no en la URL.

    Características clave:

    • Invisibilidad: Los datos no son visibles en la URL.
    • Sin límite de tamaño: Puedes enviar grandes cantidades de datos, incluyendo archivos.
    • No se guarda en el historial: No se puede guardar en marcadores directamente.
    • Seguridad: Es el único método aceptable para enviar datos sensibles.

    POST para acciones

    Usa POST siempre que la acción del formulario vaya a modificar datos en el servidor: crear un usuario, insertar un comentario, actualizar un producto, etc.

    Diagrama de Flujo de una Petición POST

    sequenceDiagram
    participant Cliente as Navegador
    participant Servidor as Servidor (PHP)

    Cliente->>Servidor: Petición POST a /login.php (Datos: user, pass en el cuerpo HTTP)
    Servidor->>Servidor: Procesa login.php
    Note right of Servidor: PHP accede a los datos<br>con la superglobal $_POST
    Servidor->>Servidor: Valida $_POST['user'] y $_POST['pass']
    Servidor-->>Cliente: Respuesta HTTP (p.ej. "Bienvenido" o "Error")

    Ejemplo 1: Formulario de Contacto

    HTML (contacto.html): 

    <form action="enviar_contacto.php" method="post">
    <p>
        <label for="email">Tu Email:</label>
        <input type="email" id="email" name="contact_email" required>
    </p>
    <p>
        <label for="mensaje">Mensaje:</label>
        <textarea id="mensaje" name="contact_message" rows="5" required></textarea>
    </p>
    <button type="submit">Enviar</button>
</form>
    PHP (enviar_contacto.php):

    <?php // PSR-12: La etiqueta de apertura de PHP debe estar en su propia línea.

declare(strict_types=1); // Buena práctica: activa el modo estricto de tipos.

// Comprobamos si el formulario fue enviado
if ($_SERVER["REQUEST_METHOD"] == "POST") {
    // Recogemos y sanitizamos las entradas
    $email = filter_input(INPUT_POST, 'contact_email', FILTER_SANITIZE_EMAIL);
    $mensaje = filter_input(INPUT_POST, 'contact_message', FILTER_SANITIZE_FULL_SPECIAL_CHARS);

    // Validamos el email después de sanitizarlo
    if (filter_var($email, FILTER_VALIDATE_EMAIL)) {
        echo "<h1>Gracias por tu mensaje</h1>";
        echo "<p>Email: {$email}</p>";
        echo "<p>Mensaje: {$mensaje}</p>";
        // Aquí iría la lógica para enviar el email
    } else {
        echo "<h1>Error: El email proporcionado no es válido.</h1>";
    }
} else {
    // Si alguien accede directamente al script sin enviar el formulario
    header("Location: contacto.html");
    exit();
}

    Ejemplo 2: Manejando Checkboxes y Radios

    HTML (encuesta.html):

    <form action="procesar_encuesta.php" method="post">
    <fieldset>
        <legend>¿Cuál es tu lenguaje de servidor preferido?</legend>
        <input type="radio" id="php" name="lenguaje_pref" value="PHP" checked> <label for="php">PHP</label><br>
        <input type="radio" id="node" name="lenguaje_pref" value="NodeJS"> <label for="node">Node.js</label><br>
        <input type="radio" id="python" name="lenguaje_pref" value="Python"> <label for="python">Python</label>
    </fieldset>

    <fieldset>
        <legend>¿Qué tecnologías de frontend usas?</legend>
        <input type="checkbox" id="html" name="tech_uso[]" value="HTML"> <label for="html">HTML</label><br>
        <input type="checkbox" id="css" name="tech_uso[]" value="CSS"> <label for="css">CSS</label><br>
        <input type="checkbox" id="js" name="tech_uso[]" value="JavaScript"> <label for="js">JavaScript</label>
    </fieldset>
    <button type="submit">Enviar Encuesta</button>
</form>

    ¡Ojo a los corchetes!

    Para los checkboxes donde el usuario puede seleccionar múltiples opciones, fíjate en name="tech_uso[]". Los corchetes [] le dicen a PHP que agrupe los valores seleccionados en un array.

    PHP (procesar_encuesta.php): 

    <?php

declare(strict_types=1);

if ($_SERVER["REQUEST_METHOD"] == "POST") {
    // Para el radio button, es un solo valor.
    $lenguaje_preferido = $_POST['lenguaje_pref'] ?? 'No especificado';
    echo "<h2>Lenguaje preferido: " . htmlspecialchars($lenguaje_preferido) . "</h2>";

    // Para los checkboxes, recibimos un array.
    if (isset($_POST['tech_uso'])) {
        $tecnologias = $_POST['tech_uso']; // Esto ya es un array
        echo "<h2>Tecnologías que usas:</h2>";
        echo "<ul>";
        foreach ($tecnologias as $tech) {
            echo "<li>" . htmlspecialchars($tech) . "</li>";
        }
        echo "</ul>";
    } else {
        echo "<h2>No has seleccionado ninguna tecnología de frontend.</h2>";
    }
}

    Reflexiona

    1. Cuando actualizas tu foto de perfil en una red social, ¿qué método HTTP se está usando y por qué?
    2. El botón "Actualizar" o "F5" del navegador repite la última petición. ¿Qué podría pasar si un usuario actualiza la página de "gracias por tu compra" que se procesó con POST? ¿Cómo podrías evitarlo? (Pista: busca sobre el patrón "Post/Redirect/Get").
    3. La superglobal $_REQUEST contiene los datos de $_GET, $_POST y $_COOKIE. ¿Por qué muchos desarrolladores consideran una mala práctica usar $_REQUEST en lugar de las más específicas $_GET y $_POST?


    Validación y Seguridad: Tu Deber como Guardián

    Nunca confíes en los datos que vienen del cliente. NUNCA. La validación en el servidor no es opcional, es una obligación.

    La única validación que cuenta es la del servidor

    La validación con HTML5 (required, type="email") o JavaScript es genial para mejorar la experiencia de usuario, pero puede ser fácilmente sorteada por un usuario malintencionado. Siempre debes validar y sanitizar todo en el lado del servidor.

    Sanitizar vs. Validar:

    • Sanitizar: Es "limpiar" la entrada. Quitar caracteres potencialmente peligrosos. Por ejemplo, eliminar etiquetas HTML de un nombre de usuario. htmlspecialchars() y filter_input() con filtros FILTER_SANITIZE_* son tus herramientas.
    • Validar: Es "comprobar" si la entrada cumple unas reglas. Por ejemplo, comprobar si un email tiene el formato correcto o si una edad es un número entre 18 y 99. filter_var() con filtros FILTER_VALIDATE_* es perfecto para esto.

    Ejemplo: Un formulario de registro robusto

    PHP (registrar.php):

    <?php

declare(strict_types=1);

$errores = [];
$nombre_usuario = '';
$email_usuario = '';

if ($_SERVER["REQUEST_METHOD"] == "POST") {
    // 1. Recoger y Sanitizar
    $nombre_usuario = filter_input(INPUT_POST, 'username', FILTER_SANITIZE_FULL_SPECIAL_CHARS);
    $email_usuario = filter_input(INPUT_POST, 'email', FILTER_SANITIZE_EMAIL);
    $edad = filter_input(INPUT_POST, 'age', FILTER_SANITIZE_NUMBER_INT);
    $password = $_POST['password'] ?? ''; // La contraseña no se sanitiza, se hashea.

    // 2. Validar
    if (empty($nombre_usuario)) {
        $errores[] = "El nombre de usuario es obligatorio.";
    }

    if (!filter_var($email_usuario, FILTER_VALIDATE_EMAIL)) {
        $errores[] = "El formato del email no es válido.";
    }

    if (!filter_var($edad, FILTER_VALIDATE_INT, ["options" => ["min_range" => 18, "max_range" => 120]])) {
        $errores[] = "Debes tener entre 18 y 120 años para registrarte.";
    }

    if (strlen($password) < 8) {
        $errores[] = "La contraseña debe tener al menos 8 caracteres.";
    }

    // 3. Procesar si no hay errores
    if (empty($errores)) {
        // Todo OK. Hashear la contraseña y guardar en la BD.
        $password_hash = password_hash($password, PASSWORD_DEFAULT);
        echo "<h1>Registro completado con éxito</h1>";
        // ...código para insertar en la base de datos...
        exit(); // Finalizamos el script para no mostrar el formulario de nuevo.
    }
}
?>
<!DOCTYPE html>
<html lang="es">
<head>
    <title>Registro</title>
    <style> .error { color: red; } </style>
</head>
<body>
    <h1>Formulario de Registro</h1>

    <?php if (!empty($errores)): ?>
        <div class="error">
            <p><strong>Por favor, corrige los siguientes errores:</strong></p>
            <ul>
                <?php foreach ($errores as $error): ?>
                    <li><?= $error ?></li>
                <?php endforeach; ?>
            </ul>
        </div>
    <?php endif; ?>

    <!-- Sticky Form: mostramos los valores anteriores -->
    <form action="registrar.php" method="post">
        <p>
            <label for="username">Nombre de Usuario:</label>
            <input type="text" name="username" id="username" value="<?= htmlspecialchars($nombre_usuario) ?>">
        </p>
        <p>
            <label for="email">Email:</label>
            <input type="email" name="email" id="email" value="<?= htmlspecialchars($email_usuario) ?>">
        </p>
        <p>
            <label for="age">Edad:</label>
            <input type="number" name="age" id="age" value="<?= isset($edad) ? htmlspecialchars($edad) : '' ?>">
        </p>
        <p>
            <label for="password">Contraseña:</label>

    File Uploads: Manejando Archivos

    Con el método POST y el enctype correcto, podemos subir archivos.

    El enctype mágico

    Para que un formulario pueda enviar archivos, debes añadir el atributo enctype="multipart/form-data" a tu etiqueta <form>. Esto le dice al navegador que divida el envío en múltiples partes (una por cada campo, y una para cada archivo).

    Los detalles del archivo subido no estarán en $_POST, sino en la superglobal $_FILES.

    La Superglobal $_FILES

    Por cada archivo subido, PHP crea una entrada en el array $_FILES. Si nuestro <input type="file"> se llama foto_perfil, accederemos a sus datos así:

    Clave Descripción Ejemplo
    $_FILES['foto_perfil']['name'] El nombre original del archivo en el ordenador del usuario. mi_foto.jpg
    $_FILES['foto_perfil']['type'] El tipo MIME del archivo, si el navegador lo proporciona. image/jpeg
    $_FILES['foto_perfil']['tmp_name'] La ruta temporal donde el servidor ha guardado el archivo. /tmp/phpA1B2C3
    $_FILES['foto_perfil']['error'] Un código de error. UPLOAD_ERR_OK (valor 0) significa que todo fue bien. 0
    $_FILES['foto_perfil']['size'] El tamaño del archivo en bytes. 123456

    Ejemplo: Subir un avatar de usuario

    HTML (subir_avatar.html): 

    <form action="procesar_avatar.php" method="post" enctype="multipart/form-data">
    <label for="avatar">Elige tu nuevo avatar (JPG, PNG, GIF):</label>
    <input type="file" id="avatar" name="user_avatar" accept="image/jpeg, image/png, image/gif">
    <button type="submit">Subir Avatar</button>
</form>

    PHP (procesar_avatar.php): 

    <?php
declare(strict_types=1);

if ($_SERVER["REQUEST_METHOD"] == "POST") {
    // 1. Comprobar que el input 'user_avatar' existe y no hay errores de subida.
    if (isset($_FILES['user_avatar']) && $_FILES['user_avatar']['error'] === UPLOAD_ERR_OK) {
        $file_tmp_path = $_FILES['user_avatar']['tmp_name'];
        $file_name = $_FILES['user_avatar']['name'];
        $file_size = $_FILES['user_avatar']['size'];
        $file_type = $_FILES['user_avatar']['type'];

        // 2. Validar la extensión y el tipo MIME
        $allowed_mime_types = ['image/jpeg', 'image/png', 'image/gif'];
        $file_extension = strtolower(pathinfo($file_name, PATHINFO_EXTENSION));
        $allowed_extensions = ['jpg', 'jpeg', 'png', 'gif'];

        if (in_array($file_type, $allowed_mime_types) && in_array($file_extension, $allowed_extensions)) {
            // 3. Validar el tamaño del archivo (ej: máximo 2MB)
            if ($file_size < 2000000) {
                // 4. Crear un nombre de archivo único para evitar colisiones
                $upload_dir = 'uploads/';
                if (!is_dir($upload_dir)) {
                    mkdir($upload_dir, 0755, true);
                }
                $new_file_name = uniqid('avatar_', true) . '.' . $file_extension;
                $dest_path = $upload_dir . $new_file_name;

                // 5. Mover el archivo de la carpeta temporal a la definitiva
                if (move_uploaded_file($file_tmp_path, $dest_path)) {
                    echo "¡Avatar subido con éxito! Guardado en: " . htmlspecialchars($dest_path);
                } else {
                    echo "Error: no se pudo mover el archivo subido.";
                }
            } else {
                echo "Error: El archivo es demasiado grande (máximo 2MB).";
            }
        } else {
            echo "Error: Tipo de archivo no permitido.";
        }
    } else {
        echo "Error en la subida. Código: " . $_FILES['user_avatar']['error'];
    }
}
?>

    No confíes en el nombre ni en el tipo

    Un atacante puede cambiar el nombre y el tipo MIME de un archivo malicioso (p.ej., virus.exe) para que parezca una imagen (foto.jpg, image/jpeg). Usa finfo_file() para verificar el tipo MIME real del contenido del archivo en el servidor como una capa extra de seguridad.

    Reflexiona

    1. ¿Por qué es una mala práctica guardar los archivos subidos con su nombre original ($file_name)?
    2. Además de la extensión, tipo y tamaño, ¿qué otra validación de seguridad podrías aplicar a un archivo de imagen subido? (Pista: piensa en imágenes que podrían contener código malicioso).
    3. En un entorno de producción, las directivas de php.ini como upload_max_filesize y post_max_size son cruciales. ¿Qué problema ocurriría si post_max_size es más pequeño que upload_max_filesize?

      • ¿Hay algo que tú hiciste diferente? ¿Por qué?
      • ¿Aprendiste alguna técnica nueva de su enfoque?
    Aplicación en el Mundo Real:

    Las revisiones de código (Code Reviews) son una práctica estándar en los equipos de desarrollo profesionales. Sirven para mejorar la calidad del código, compartir conocimiento y, crucialmente, detectar bugs y fallos de seguridad antes de que lleguen a producción.
