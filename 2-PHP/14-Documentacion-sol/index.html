    PHP - Documentación: El Arte de Dejar Huellas Útiles

    Bueno, hoy vamos a hablar de algo tan crucial como el código mismo: la documentación. Si programar es construir una casa, documentar es dejar los planos, el manual de usuario y las etiquetas en la caja de fusibles. Sin eso, la casa es una trampa mortal para el siguiente que intente hacer una reforma... que muy probablemente, serás tú mismo dentro de seis meses.

    La analogía principal de hoy es simple: documentar tu código es como ser un cartógrafo de tu propia aplicación. Estás creando un mapa detallado para que otros (o tu "yo" del futuro, que tendrá la memoria de un pez) puedan navegar por la complejidad de tu creación sin perderse. En el mundo laboral, un proyecto sin documentación no es un "desafío emocionante", es una "patata caliente" que nadie quiere tocar. ¡Vamos a aprender a dibujar mapas claros y útiles!

    Conceptos Fundamentales

    Antes de ponernos a dibujar, aclaremos algunos términos clave.

    Definición: Documentación de Código

    Es el conjunto de textos y comentarios integrados en el código fuente que explican su estructura, funcionamiento, propósito y uso. No es un manual de usuario para el cliente final, sino un manual técnico para desarrolladores.

    Definición: DocBlock

    Un DocBlock es un tipo especial de comentario en PHP que empieza por /** y termina con */. Todo lo que está dentro se considera una descripción formal de la pieza de código que le sigue (una función, una clase, una propiedad...). Es la unidad básica de nuestra cartografía.

    Definición: phpDocumentor

    Es la herramienta estándar en el ecosistema PHP para procesar automáticamente todos tus DocBlocks y generar un sitio web navegable con la documentación completa de tu API. Es como un dron que fotografía todo tu mapa y lo convierte en un Google Maps interactivo de tu proyecto.

    Definición: Anotación (o Tag)

    Son palabras clave especiales que usamos dentro de un DocBlock, siempre precedidas por una @ (arroba). Sirven para estructurar la información de forma estándar. Ejemplos son @param para describir un parámetro de una función o @return para describir lo que devuelve.


    Desarrollo y Ejemplos Prácticos

    ¿Por qué documentar? El mapa del tesoro de tu código

    A ver, seamos sinceros. Cuando terminas una funcionalidad, lo último que apetece es escribir. Quieres pasar a lo siguiente, a ese nuevo reto que te quita el sueño. Pero, ¡alto ahí! Imagina que te unes a un proyecto y te dan un código de 50.000 líneas sin un solo comentario. Es como si te dijeran "el tesoro está en esa isla" y te dieran un dibujo de una palmera hecho en una servilleta. Frustrante, ¿verdad?

    Documentar bien te ahorra tiempo y dinero a largo plazo:

    • Facilita la colaboración: Permite que otros entiendan tu código sin tener que preguntarte cada cinco minutos.
    • Simplifica el mantenimiento: Cuando algo se rompe, el mapa te dice exactamente dónde está la tubería rota.
    • Es un regalo para tu "yo" futuro: Te sorprendería lo rápido que olvidamos por qué tomamos ciertas decisiones.
    • Genera confianza: Un API bien documentada es profesional y fácil de usar para otros desarrolladores que quieran integrarse con tu aplicación.
    Sin Documentación (El caos) Con Buena Documentación (El orden)
    Onboarding de nuevos devs: Semanas, con frustración. Onboarding de nuevos devs: Días, con autonomía.
    Tasa de errores: Alta. Cambiar algo aquí rompe algo allá. Tasa de errores: Baja. Se entienden las dependencias.
    Mantenibilidad: Un infierno. Nadie quiere tocar el código "legacy". Mantenibilidad: Sencilla. El código es predecible.
    Reutilización de código: Nula. Es más fácil reescribir que entender. Reutilización de código: Alta. Se sabe qué hace cada pieza.

    Reflexiona

    1. ¿Alguna vez has intentado montar un mueble de IKEA sin instrucciones? ¿Cómo te sentiste? Compara esa experiencia con intentar entender un código sin documentar.
    2. Piensa en la aplicación o web que más usas. ¿Crees que sus desarrolladores tendrán una buena documentación interna? ¿Por qué?
    3. ¿Qué es peor: un código brillante pero indescifrable, o un código más simple pero perfectamente documentado? Debate con tus compañeros.


    phpDocumentor: Tu Generador de Mapas Automático

    Ya que estamos convencidos de la importancia de documentar, vamos a ver la herramienta que nos hará la vida más fácil. phpDocumentor lee tus comentarios (DocBlocks) y genera un sitio web navegable con toda la información.

    Instalación

    Hay dos vías principales, la de "voy a probarlo rápido" y la "así se hace en el mundo real".

    1. Como binario (La vía rápida): Te descargas un archivo, le das permisos y lo mueves a un sitio donde tu sistema lo encuentre. 

      wget https://phpdoc.org/phpDocumentor.phar
chmod +x phpDocumentor.phar
sudo mv phpDocumentor.phar /usr/local/bin/phpdoc
phpdoc --version

    2. Con Docker (La vía profesional): !!! tip inline "Docker: El Contenedor Mágico" Docker nos permite ejecutar aplicaciones en un entorno aislado y controlado. Usar la imagen oficial de phpDocumentor garantiza que funciona en cualquier máquina, sin conflictos de versiones de PHP o dependencias. Es el estándar en la industria.

      El comando base es: docker run --rm -v "$(pwd)":/data phpdoc/phpdoc:3. Esto crea un contenedor, mapea tu directorio actual a la carpeta /data dentro del contenedor, y ejecuta phpDocumentor. Al terminar, el contenedor se autodestruye (--rm), dejando todo limpio.

    Ejecución

    Una vez instalado, el uso es sencillo. Le dices qué carpeta analizar (-d de directory) y dónde dejar el resultado (-t de target).

    # Versión con el binario
# Suponiendo que nuestro código está en 'src/' y queremos la doc en 'docs/api'
phpdoc -d ./src -t docs/api

# Versión con Docker
docker run --rm -v "$(pwd)":/data phpdoc/phpdoc:3 -d ./src -t ./docs/api

    Después de ejecutarlo, abre el archivo docs/api/index.html en tu navegador y ¡magia! Tendrás tu documentación lista para explorar.

    La Sintaxis de la Sabiduría: DocBlocks y Anotaciones

    Aquí está el corazón del asunto. Un DocBlock bien escrito es una obra de arte. Su estructura es clave.

    <?php
/**
 * Sumario conciso de lo que hace el elemento (una línea).
 *
 * Descripción más detallada. Puede ocupar varias líneas y explicar
 * el porqué, el contexto o dar ejemplos de uso. Es opcional pero
 * muy recomendable para lógica compleja.
 *
 * @anotacion1 valor descripción
 * @anotacion2 valor descripción
 */

    Anotaciones Esenciales

    Anotación Uso y Ejemplo
    @author Indica el autor. @author Nombre Apellido <email@ejemplo.com>
    @package Agrupa clases en "paquetes" lógicos, usando el namespace. @package App\Model
    @var Describe una propiedad de una clase. @var string El nombre del usuario.
    @param Describe un parámetro de una función/método. @param int $userId El ID del usuario a buscar.
    @return Describe el valor que devuelve una función/método. @return User|null El objeto User o null si no se encuentra.
    @throws Indica que el método puede lanzar una excepción. @throws \InvalidArgumentException Si el ID es negativo.

    Ejemplos Prácticos

    Ejemplo 1: Una función sencilla (PSR-12)

    <?php

declare(strict_types=1);

namespace App\Utils;

/**
 * Calcula la suma de dos números enteros.
 *
 * Esta función toma dos enteros y devuelve su suma. Se asegura
 * de que el resultado también sea un entero.
 *
 * @param int $a El primer sumando.
 * @param int $b El segundo sumando.
 *
 * @return int La suma de $a y $b.
 */
function sumar(int $a, int $b): int
{
    return $a + $b;
}

    Ejemplo 2: Una clase más completa

    Vamos a mejorar el ejemplo de la clase Cliente del material original, aplicando PSR-12 y añadiendo más detalles.

    <?php

declare(strict_types=1);

namespace App\Videoclub\Model;

use App\Videoclub\Excepciones\ClienteException;
use App\Videoclub\Excepciones\SoporteYaAlquiladoException;

/**
 * Representa a un cliente del videoclub.
 *
 * Gestiona el alquiler y devolución de soportes (películas, juegos...).
 * Cada cliente tiene un número único y una lista de productos que
 * tiene actualmente en su poder.
 *
 * @package App\Videoclub\Model
 * @author Aitor Medrano <a.medrano@edu.gva.es>
 */
class Cliente
{
    /**
     * Colección de soportes que el cliente tiene alquilados.
     *
     * @var Soporte[]
     */
    private array $soportesAlquilados = [];

    /**
     * @param string $nombre El nombre completo del cliente.
     * @param string $numero El identificador único del cliente.
     */
    public function __construct(
        public string $nombre,
        private string $numero
    ) {
    }

    /**
     * Añade un soporte a la lista de alquilados del cliente.
     *
     * @param Soporte $soporte El soporte a alquilar.
     *
     * @return void
     * @throws SoporteYaAlquiladoException si el cliente ya tiene este soporte.
     */
    public function alquilar(Soporte $soporte): void
    {
        if ($this->tieneAlquilado($soporte)) {
            throw new SoporteYaAlquiladoException("El cliente ya tiene este soporte alquilado.");
        }
        $this->soportesAlquilados[] = $soporte;
    }

    /**
     * Comprueba si un soporte específico ya está en la lista de alquilados.
     *
     * @param Soporte $soporte El soporte a comprobar.
     *
     * @return bool True si ya lo tiene, false en caso contrario.
     */
    public function tieneAlquilado(Soporte $soporte): bool
    {
        return in_array($soporte, $this->soportesAlquilados, true);
    }
}

    Más Allá de lo Básico: Anotaciones Avanzadas

    Hay más vida más allá de @param y @return. Estas anotaciones te dan superpoderes para crear una documentación mucho más rica y conectada.

    • @see: Hace referencia a otra clase, método o URL que esté relacionada. Es genial para decir "si esto te interesa, mira también esto otro".
    • @link: Similar a @see pero crea un enlace web a una URL externa.
    • @version: Para indicar la versión del proyecto o paquete en la que se introdujo el elemento.
    • @deprecated: Marca un método o propiedad como obsoleto. ¡Importantísimo! Indica que ya no se debería usar y, a menudo, sugiere una alternativa.

/**

 * @see \App\Repositorio\ClienteRepositorio::buscarClientesPorCriterio()
 */
public function buscarClientes(string $query): array
{
    // ... lógica antigua ...
}

    Aplicación en el Mundo Real

    No estamos aprendiendo esto para aprobar un examen. Lo estáis aprendiendo porque así es como funcionan las empresas de desarrollo serias.

    • Frameworks Open Source (Symfony, Laravel, etc.): La totalidad de su código está documentada con este estándar. Es lo que permite que millones de desarrolladores en todo el mundo puedan usarlos, entenderlos y contribuir a ellos. Su documentación de API se genera directamente de estos comentarios.
    • APIs Públicas (Stripe, Twilio, etc.): Cuando una empresa ofrece una API para que otros se integren, la calidad de su documentación es un factor de venta decisivo. Una API potente pero mal documentada está condenada al fracaso. Utilizan herramientas como phpDocumentor para generar portales para desarrolladores atractivos y funcionales.
    • Equipos de Desarrollo: Dentro de una empresa, el equipo de backend crea una API. El equipo de frontend y el de mobile necesitan consumir esa API. En lugar de reuniones constantes y preguntas por Slack, los equipos de frontend y mobile simplemente consultan la documentación generada. Es el contrato que define cómo se comunican las diferentes partes de la aplicación.

    En resumen, saber documentar es una skill tan importante como saber escribir código. Te posiciona como un desarrollador profesional, limpio y que sabe trabajar en equipo.

    EJERCICIOS

    Ejercicios de Consolidación

    1) El Código Olvidado: Rescatando la Documentación | Nivel: Fácil

    Objetivo:

    Identificar y corregir errores en la documentación de una clase PHP existente para que cumpla con los estándares de phpDocumentor y refleje fielmente la funcionalidad del código.

    Setup Inicial:

    Formas parte del equipo de mantenimiento de "InnovateSoft". Te han asignado una tarea para limpiar el código de un módulo antiguo antes de integrarlo en el nuevo SDK de la compañía. El siguiente código funciona, pero su documentación es un desastre.

    <?php
// Fichero: GestorCache.php

declare(strict_types=1);

namespace InnovateSoft\Utils;

use \Exception;

/**
 * Una clase para manejar la caché.
 * @package InnovateSoft
 */
class GestorCache
{
    /**
     * @param int $ttl
     */
    private array $storage = [];
    private int $defaultTtl = 3600;

    /**
     * @return void
     */
    public function __construct(int $defaultTtl)
    {
        $this->defaultTtl = $defaultTtl;
    }

    /**
     * Añade un valor a la caché.
     *
     * @param string $key
     * @param string $value
     * @return string
     */
    public function set(string $key, $value): bool
    {
        $this->storage[$key] = [
            'value' => $value,
            'expires' => time() + $this->defaultTtl
        ];
        return true;
    }

    /**
     * Obtiene un valor de la caché.
     *
     * @param int $key
     * @throws Exception
     */
    public function get(string $key): mixed
    {
        if (!isset($this->storage[$key])) {
            return null;
        }

        if (time() > $this->storage[$key]['expires']) {
            unset($this->storage[$key]);
            return null;
        }

        return $this->storage[$key]['value'];
    }
}

    1. Copia el código anterior en un fichero llamado GestorCache.php.
    2. Identifica todos los errores y omisiones en los DocBlocks. Presta atención a los tipos de datos, descripciones, anotaciones faltantes o incorrectas y el formato general.
    3. Corrige la documentación para que sea precisa, completa y siga las buenas prácticas vistas en el tema.
    Aplicación en el Mundo Real:

    El "código heredado" (legacy code) es extremadamente común. A menudo, la primera tarea al modernizar un sistema es entender y mejorar su documentación para poder modificarlo con seguridad. Esta habilidad es fundamental para cualquier desarrollador de mantenimiento o que se incorpore a un proyecto ya existente.

    VER Solución. 
    <?php
// Fichero: GestorCache.php

declare(strict_types=1);

namespace InnovateSoft\Utils;

/**
 * Gestiona un sistema de caché simple en memoria.
 *
 * Esta clase permite almacenar y recuperar valores asociándolos a una clave
 * y con un tiempo de vida (TTL) determinado.
 *
 * @package InnovateSoft\Utils
 * @author Tu Nombre <tu.email@innovatesoft.com>
 */
class GestorCache
{
    /**
     * Almacén interno para los datos de la caché.
     * @var array
     */
    private array $storage = [];

    /**
     * Tiempo de vida por defecto para las entradas de caché, en segundos.
     * @var int
     */
    private int $defaultTtl;

    /**
     * Constructor de la clase GestorCache.
     *
     * @param int $defaultTtl El tiempo de vida (TTL) por defecto para las nuevas entradas.
     */
    public function __construct(int $defaultTtl = 3600)
    {
        $this->defaultTtl = $defaultTtl;
    }

    /**
     * Añade un valor a la caché con una clave.
     *
     * @param string $key   La clave única para el elemento.
     * @param mixed  $value El valor a almacenar.
     *
     * @return bool Devuelve siempre true.
     */
    public function set(string $key, mixed $value): bool
    {
        $this->storage[$key] = [
            'value' => $value,
            'expires' => time() + $this->defaultTtl
        ];
        return true;
    }

    /**
     * Obtiene un valor de la caché.
     *
     * Si la clave no existe o ha expirado, devuelve null.
     *
     * @param string $key La clave del elemento a recuperar.
     *
     * @return mixed El valor almacenado o null si no se encuentra o ha expirado.
     */
    public function get(string $key): mixed
    {
        if (!isset($this->storage[$key])) {
            return null;
        }

        if (time() > $this->storage[$key]['expires']) {
            unset($this->storage[$key]);
            return null;
        }

        return $this->storage[$key]['value'];
    }
}

    <?php
// Fichero: ApiResponse.php

declare(strict_types=1);

namespace InnovateSoft\Http;

use \JsonSerializable;

/**
 * Representa una respuesta HTTP recibida de una API.
 *
 * Esta clase encapsula el cuerpo, el código de estado y las cabeceras
 * de una respuesta HTTP, proporcionando métodos para acceder a ellos de forma estructurada.
 *
 * @package InnovateSoft\Http
 * @author Tu Nombre <tu.email@innovatesoft.com>
 */
class ApiResponse implements JsonSerializable
{
    /**
     * Cabeceras de la respuesta HTTP.
     * @var array<string, string>
     */
    private array $headers;

    /**
     * Crea una nueva instancia de ApiResponse.
     *
     * @param string $body El cuerpo de la respuesta, generalmente en formato JSON.
     * @param int $statusCode El código de estado HTTP (ej: 200, 404, 500).
     * @param array<string, string> $headers Cabeceras adicionales de la respuesta.
     */
    public function __construct(
        private string $body,
        private int $statusCode,
        array $headers = []
    ) {
        $this->headers = array_merge(['Content-Type' => 'application/json'], $headers);
    }

    /**
     * Devuelve el cuerpo de la respuesta en formato raw (string).
     *
     * @return string El cuerpo de la respuesta.
     */
    public function getBody(): string
    {
        return $this->body;
    }

    /**
     * Devuelve el código de estado HTTP.
     *
     * @return int El código de estado.
     */
    public function getStatusCode(): int
    {
        return $this->statusCode;
    }

    /**
     * Decodifica el cuerpo JSON de la respuesta y lo devuelve como un array asociativo.
     *
     * @return array|null Un array asociativo con los datos o null si el JSON es inválido.
     */
    public function getDecodedBody(): ?array
    {
        $data = json_decode($this->body, true);
        return json_last_error() === JSON_ERROR_NONE ? $data : null;
    }

    /**
     * Especifica los datos que deben ser serializados a JSON.
     *
     * @return mixed Los datos que pueden ser serializados por json_encode().
     */
    public function jsonSerialize(): mixed
    {
        return [
            'statusCode' => $this->statusCode,
            'headers' => $this->headers,
            'body' => $this->getDecodedBody()
        ];
    }
}

    VER Solución. 
    <?php
// Fichero: Logger.php

declare(strict_types=1);

namespace InnovateSoft\Logging;

/**
 * Una clase de logging simple que escribe en un fichero.
 *
 * Este Logger proporciona una forma básica de registrar mensajes de diferentes
 * niveles en un fichero de texto plano. Sigue los principios del estándar PSR-3.
 *
 * @package InnovateSoft\Logging
 * @author Tu Nombre <tu.email@innovatesoft.com>
 * @version 1.1.0
 * @link https://www.php-fig.org/psr/psr-3/ Estándar de Logger Interface.
 */
class Logger
{
    /**
     * La ruta completa al fichero donde se guardarán los logs.
     * @var string
     */
    private string $logFilePath;

    /**
     * Crea una instancia del Logger.
     *
     * @param string $logFilePath La ruta al fichero de log. Se creará si no existe.
     */
    public function __construct(string $logFilePath)
    {
        $this->logFilePath = $logFilePath;
    }

    /**
     * Escribe un mensaje de log general.
     *
     * @param string $message El mensaje a registrar.
     * @return void
     */
    public function log(string $message): void
    {
        $this->write("[LOG]", $message);
    }

    /**
     * Escribe un mensaje de información.
     *
     * @param string $message El mensaje informativo a registrar.
     * @return void
     */
    public function info(string $message): void
    {
        $this->write("[INFO]", $message);
    }

    /**
     * Escribe un mensaje de error.
     *
     * @param string $message El mensaje de error a registrar.
     * @return void
     */
    public function error(string $message): void
    {
        $this->write("[ERROR]", $message);
    }

    /**
     * Método antiguo para añadir un log.
     *
     * @param string $message El mensaje a registrar.
     * @return void
     *
     * @deprecated 1.1.0 Utilizar el método log() en su lugar.
     * @see self::log()
     */
    public function addLog(string $message): void
    {
        $this->log($message);
    }

    /**
     * Escribe el mensaje formateado en el fichero de log.
     *
     * @param string $level   El nivel del log (ej: [INFO], [ERROR]).
     * @param string $message El mensaje.
     */
    private function write(string $level, string $message): void
    {
        $formattedMessage = sprintf(
            "[%s] %s: %s" . PHP_EOL,
            date('Y-m-d H:i:s'),
            $level,
            $message
        );

        file_put_contents($this->logFilePath, $formattedMessage, FILE_APPEND);
    }
}
